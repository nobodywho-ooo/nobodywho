# NobodyWho Documentation

Built with [Docusaurus 3](https://docusaurus.io/). Each binding (Kotlin, Python, Swift, React Native, Flutter, Godot) has its own docs instance with independent versioning.

## Development

```bash
cd docs
npm install
npm start        # Dev server with hot reload
npm run build    # Production build (must pass with zero errors)
npm run serve    # Serve the production build locally
```

## Project Structure

```
docs/
  docs/                  # Shared docs (Overview, LLM Basics, Model Selection)
  docs-kotlin/           # Kotlin binding docs
  docs-python/           # Python binding docs
  docs-swift/            # Swift binding docs
  docs-react-native/     # React Native binding docs
  docs-flutter/          # Flutter binding docs
  docs-godot/            # Godot binding docs
  sidebars/              # Sidebar config for each binding
  src/                   # Theme overrides, components, CSS
  static/                # Images, fonts, favicon
  plugins/               # Custom plugins (llms.txt generator)
  docusaurus.config.ts   # Main site configuration
```

## Editing Docs

Edit markdown files directly in the `docs-<binding>/` directories. Changes are reflected immediately with `npm start`.

## Versioning

Each binding has three kinds of docs:

- **Latest tagged release** — the default. Lives in `<binding>_versioned_docs/version-<v>/`. Served at `/<binding>/`.
- **`main`** — the unreleased state from the `main` branch. Lives in `docs-<binding>/`. Served at `/<binding>/main/` with an "unreleased" banner.
- **Older releases** — previous snapshots. Served at `/<binding>/<v>/` with an "unmaintained" banner.

The "latest release" per binding is declared in `docusaurus.config.ts`:

```ts
const latestReleases: Record<string, string> = {
  kotlin: '2.0.0',
  python: '1.5.0',
  swift: '2.1.0',
  'react-native': '2.3.0',
  flutter: '2.3.0',
  godot: '9.4.0',
};
```

The banner on each non-default version is chosen automatically by Docusaurus based on whether the version is newer or older than `lastVersion` (which is set from `latestReleases`).

### Cutting docs for a new release

When you tag a new release (e.g. `nobodywho-python-v1.5.0`), snapshot the current `main`-branch docs as the new release and bump the default:

**1. Snapshot `docs-python/` at the new version:**

```bash
cd docs
npx docusaurus docs:version:python 1.5.0
```

This creates:

```
python_versioned_docs/version-1.5.0/    # Frozen copy of docs-python/
python_versioned_sidebars/              # Frozen sidebar
python_versions.json                    # ["1.5.0", "1.4.0", ...]
```

**2. Update `latestReleases` in `docusaurus.config.ts`:**

```ts
python: '1.5.0',  // was '1.4.0'
```

**3. Commit** the versioned folders, the versions JSON, and the config change.

After building:
- `/python/` — 1.5.0 (the new default, no banner)
- `/python/main/` — `docs-python/`, banner "unreleased"
- `/python/1.4.0/` — previous release, banner "unmaintained"

If you want to start landing docs changes for the *next* release on `main`, edit `docs-python/` directly — those changes show up under `/python/main/` until the next snapshot.

The same workflow applies to any binding — replace `python` with `swift`, `flutter`, `react-native`, or `godot`.

### Removing an old version

Delete the versioned files and update the versions list:

```bash
rm -rf python_versioned_docs/version-1.4.0
rm python_versioned_sidebars/version-1.4.0-sidebars.json
```

Edit `python_versions.json` and remove `"1.4.0"` from the array. If no versions remain, delete the file entirely. The sidebar dropdown disappears automatically when only one version exists.

## Deployment

The site deploys to Cloudflare Pages via `.github/workflows/docs.yml`. It triggers on push to `main`. All versioned docs are included in the static build — no server-side logic needed.

## LLM-Friendly Output

The build generates two files:
- `build/llms.txt` — page index with links
- `build/llms-full.txt` — full concatenated documentation

These are generated by `plugins/llms-txt/index.js` during the `postBuild` step.
