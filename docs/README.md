# NobodyWho Documentation

Built with [Docusaurus 3](https://docusaurus.io/). Each binding (Python, Swift, React Native, Flutter, Godot) has its own docs instance with independent versioning.

## Development

```bash
cd docs
npm install
npm start        # Dev server with hot reload
npm run build    # Production build (must pass with zero errors)
npm run serve    # Serve the production build locally
```

## Project Structure

```
docs/
  docs/                  # Shared docs (Overview, LLM Basics, Model Selection)
  docs-python/           # Python binding docs
  docs-swift/            # Swift binding docs
  docs-react-native/     # React Native binding docs
  docs-flutter/          # Flutter binding docs
  docs-godot/            # Godot binding docs
  sidebars/              # Sidebar config for each binding
  src/                   # Theme overrides, components, CSS
  static/                # Images, fonts, favicon
  plugins/               # Custom plugins (llms.txt generator)
  docusaurus.config.ts   # Main site configuration
```

## Editing Docs

Edit markdown files directly in the `docs-<binding>/` directories. Changes are reflected immediately with `npm start`.

## Versioning

Each binding has its own version history. The current version labels are set in `docusaurus.config.ts`:

```ts
const bindingVersions: Record<string, string> = {
  python: '1.4.0',
  swift: '2.0.0',
  'react-native': '2.2.0',
  flutter: '2.2.0',
  godot: '9.3.0',
};
```

### Updating docs for a new release (no breaking changes)

If the docs content hasn't changed significantly, just update the version label in `docusaurus.config.ts`:

```ts
python: '1.5.0',  // was '1.4.0'
```

No snapshot needed. The version badge on each page updates automatically.

### Updating docs for a breaking release (preserve old docs)

When a release has breaking API changes and you want users on the old version to still access the old docs:

**1. Snapshot the current docs before making changes:**

```bash
cd docs
npx docusaurus docs:version:python 1.4.0
```

This creates a frozen copy:

```
python_versioned_docs/version-1.4.0/    # Full copy of docs-python/
python_versioned_sidebars/              # Frozen sidebar
python_versions.json                    # ["1.4.0"]
```

**2. Update the docs in `docs-python/`** to reflect the new API.

**3. Update the version label** in `docusaurus.config.ts`:

```ts
python: '1.5.0',
```

**4. Commit everything** — the versioned folders, the updated docs, and the config change.

After building, the site will have:
- `/python/` — version 1.5.0 (from `docs-python/`)
- `/python/1.4.0/` — version 1.4.0 (from `python_versioned_docs/version-1.4.0/`)

A version dropdown appears automatically in the left sidebar when multiple versions exist.

The same workflow applies to any binding — replace `python` with `swift`, `flutter`, `react-native`, or `godot`.

### Removing an old version

Delete the versioned files and update the versions list:

```bash
rm -rf python_versioned_docs/version-1.4.0
rm python_versioned_sidebars/version-1.4.0-sidebars.json
```

Edit `python_versions.json` and remove `"1.4.0"` from the array. If no versions remain, delete the file entirely. The sidebar dropdown disappears automatically when only one version exists.

## Deployment

The site deploys to Cloudflare Pages via `.github/workflows/docs.yml`. It triggers on push to `main`. All versioned docs are included in the static build — no server-side logic needed.

## LLM-Friendly Output

The build generates two files:
- `build/llms.txt` — page index with links
- `build/llms-full.txt` — full concatenated documentation

These are generated by `plugins/llms-txt/index.js` during the `postBuild` step.
